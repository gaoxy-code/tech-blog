# blog-writing

このスキルは `src/posts/*.md` を扱うときの規約・テンプレ・チェックリストを定義する。
新規作成と既存記事の手入れ (添削 / 校閲 / 公開準備) で共通して使う。

`/blog:new` `/blog:review` `/blog:proofread` `/blog:publish` の各コマンドはこの SKILL.md を読み、対応するチェックリストを実行する薄いラッパー。自然言語で「記事を書いて」「レビューして」と頼まれたときも同じルールを適用する。

## ファイルの場所と命名

- 記事は `src/posts/<slug>.md` に置く
- `<slug>` は kebab-case、日付プレフィックスは付けない (例: `gas-line-messaging-api.md`)
- ファイル名はタイトルの英訳ではなく、主要トピックを表すキーワードの並び (例: `svelte5-runes-basics`)

## フロントマター仕様

```yaml
---
title: タイトル
description: 説明 (1〜2 文。検索結果や OG に出る)
date: YYYY-MM-DD
tags:
  - tag-a
  - tag-b
---
```

ルール:

- 4 項目固定: `title` / `description` / `date` / `tags`
- `tags` は **2 スペースインデントの YAML リスト**。インデントが崩れると次のキーが配列要素として食われる
- `date` は `YYYY-MM-DD` 形式。記事の初回作成日。コマンド経由で生成するなら今日の日付を入れる
- 公開記事に `draft: true` を残さない。下書き中は `description:` の次に `draft: true` を入れて良い (`getAllPosts()` が除外する)
- `tags` の語彙は `src/posts/*.md` 既存タグと揃える。新しい技術を導入するときだけ新タグ追加

下書き状態の許容形:

```yaml
---
title: タイトル
description: 説明
date: 2026-04-30
draft: true
tags:
  - tag-a
---
```

## 構成テンプレ (使い分け)

| 種別                                 | 使う場面                                 | 骨格                                                                 | お手本                                 |
| ------------------------------------ | ---------------------------------------- | -------------------------------------------------------------------- | -------------------------------------- |
| 解説型 (`templates/concept.md`)      | 概念・API・記法を解説する                | 導入 → 主要トピック 3〜4 個 → まとめ表 → 参照                        | `src/posts/svelte5-runes-basics.md`    |
| 手順記事型 (`templates/tutorial.md`) | セットアップ手順・「動くまで」を再現する | 構成図 → 事前準備 → 実装 → デプロイ → ハマりポイント → まとめ → 参照 | `src/posts/cloudflare-pages-deploy.md` |

迷ったら解説型を使う。手順記事型は再現に外部サービスや CLI が絡むケース限定。

## 全記事に共通の構成ルール

1. **冒頭は導入段落 1〜2 段落**。`## はじめに` セクションは置かない。1 行目から本文に入る
2. セクション区切りは `## ` の見出しのみ。`---` 水平線は使わない (mdsvex の hr で見栄えが安定しないので避ける)
3. **末尾は必ず `## まとめ` → `## 参照` の順** で締める。CLAUDE.md の規約
4. `## 参考リンク` `## 参考` `## References` は使わない。**`## 参照`** で統一
5. コードブロックには言語タグを必ず付ける (` ```ts ` / ` ```svelte ` / ` ```bash ` / ` ```json ` / ` ```yaml ` / ` ```javascript ` など)
6. 表は `|` で区切る GitHub flavored markdown。比較・対応表・イベント一覧などで活用

## `## 参照` のルール

CLAUDE.md の規約をそのまま適用:

- 必須: 主題ライブラリ・ツールの **公式サイト** または **公式ドキュメント**
- 推奨: 公式 GitHub リポジトリ、仕様 / RFC、関連する公式ガイド
- リンクテキストは「サイト名 + 種別」(例: `Svelte Runes ドキュメント`、`clasp GitHub リポジトリ`)
- **個人ブログ・Qiita・Zenn 等の非公式記事は載せない** (一次情報を優先)
- 主題が複数ある記事は `### 〜` のサブセクションでグルーピングして良い (例: `gas-line-messaging-api.md` の LINE / GAS / clasp)
- 最低 3 件は載せる。1 件しか思いつかないなら参照不足のサインで、執筆中の事実確認も足りていない可能性が高い

## 日本語スタイル

- です/ます調で統一
- **半角英数と日本語の境目に半角スペース**を 1 つ入れる (例: `GAS と LINE Messaging API`)
- 句読点は「、」「。」を使う
- 主語の省略は読みづらくならない範囲で
- 冗長な「〜することができます」より「〜できます」を選ぶ
- 「私」「弊社」「皆さん」のような書き手・読み手の人称代名詞は基本不要
- 絵文字は本文中で多用しない。表や Tips の強調で 1 記事あたり数個までが目安。`📌` `⚠️` `📸` などを多重に使うと装飾過多になるので削る

## スクショ placeholder の扱い

手順記事型で実装中に貼り出す形式 (執筆フェーズ):

```markdown
<!-- スクショ: 〇〇画面 -->

> 📸 **スクショ① 〇〇画面の説明** (値はマスク推奨)
```

ルール:

- 下書き (`draft: true`) の間は残して良い
- `## 参照` の手前まで進んだら、`/blog:publish` 実行時にユーザーへ「実画像に差し替えるか、それとも全削除するか」を必ず確認する
- 公開記事に placeholder を残したまま `draft: true` を外さない

## 画像ファイルの管理

実画像は以下のルールで配置・参照する:

### 配置先

```
static/posts/<slug>/<NN>-<short-description>.<ext>
```

- `<slug>` は記事ファイル名と揃える (例: `gas-line-messaging-api`)
- `<NN>` は 2 桁ゼロ埋めの連番 (`01`, `02`, ...)。Markdown 内のスクショ番号 (① ② ...) と一致させる
- `<short-description>` は kebab-case の英数字 (例: `console-top`, `qr-code`, `permissions-dialog`)
- 拡張子は基本 `.png`、写真系は `.jpg`、動きの説明は `.gif`、ベクタは `.svg`

例:

```
static/posts/gas-line-messaging-api/
├── 01-developers-console-top.png
├── 02-channel-create-form.png
├── 03-access-token-issued.png
└── 11-line-notification.png
```

### Markdown での参照書式

```markdown
![Messaging API 設定タブのアクセストークン発行画面](/posts/gas-line-messaging-api/03-access-token-issued.png)
```

- 絶対パス (`/posts/...`) で書く。`static/` プレフィックスは不要 (Cloudflare Pages が `static/` を root として配信)
- alt テキストは「何が写っているか」を簡潔に。画像が表示できない読者・スクリーンリーダー向けの情報源
- 旧 placeholder (`<!-- スクショ: ... -->` と `> 📸 ...` の 2 行) はこの 1 行で置き換える

### `/blog:publish` での差し替え手順

1. `static/posts/<slug>/` ディレクトリを作成 (無ければ)
2. ユーザーが用意した画像ファイルをディレクトリへ配置 (このステップはユーザー作業)
3. Markdown 内の `<!-- スクショ: 〇〇 -->` と `> 📸 **スクショ① 〇〇** (...)` の 2 行ペアを、対応する `![alt](/posts/<slug>/NN-...png)` 1 行に置換
4. 番号は placeholder の番号 (① ② ...) を `NN` 連番にマッピング

### 画像を使わない場合

placeholder を全部削除してしまって構わない。「画像が無くても文章で再現できる」が手順記事型の理想。ユーザーが「画像不要」を選んだら 2 行ペアごと丸ごと削除する。

## ファイル名 slug の決め方

1. 主要なライブラリ / 技術名から始める (`svelte5-`, `claude-code-`, `cloudflare-`, `gas-`)
2. その下に主題を続ける (`-runes-basics`, `-hooks`, `-pages-deploy`, `-line-messaging-api`)
3. 既存記事に同系統があればプレフィックスを揃える

## 添削チェックリスト (`/blog:review`)

対象記事を 1 つずつ点検する。問題があれば「行番号 + 何が違反か + 修正案」を提示する。

- [ ] **フロントマター 4 項目が揃っている** (`title` / `description` / `date` / `tags`)
- [ ] **`tags` のインデントが 2 スペース**で、後続キーを巻き込んでいない
- [ ] `date` が `YYYY-MM-DD` 形式
- [ ] 冒頭に `## はじめに` セクションが無い (あるなら導入段落に格下げ)
- [ ] **末尾が `## まとめ` → `## 参照` の順**で並んでいる
- [ ] **`## 参照` という見出し名**になっている (`## 参考リンク` 等は不可)
- [ ] `## 参照` に公式情報のリンクが最低 3 件
- [ ] `## 参照` に個人ブログ等の非公式リンクが混ざっていない
- [ ] 文中の `---` 水平線が無い (もしくは意味のある 1〜2 箇所だけ)
- [ ] コードブロック全てに言語タグが付いている
- [ ] スクショ placeholder と `draft: true` の整合 (placeholder があるのに draft が外れていないか)
- [ ] 半角英数と日本語の境にスペースがある
- [ ] 装飾絵文字 (`📌` `⚠️` 等) が多すぎないか (記事全体で 5 個以内が目安)

## 校閲チェックリスト (`/blog:proofread`)

文章レベルの精査。違反箇所と修正案を提示する。

- [ ] 誤字脱字・送り仮名のゆれ (例: `行なう` / `行う`)
- [ ] 同じ言い回しの連続 (3 段落以上連続で「〜できます。」が続く等)
- [ ] 冗長表現 (`〜することができます` / `〜という形で` / `〜について` の連発)
- [ ] 主語と述語のねじれ
- [ ] 半角と全角の混在 (例: `URL` と `ＵＲＬ`)
- [ ] 数値の単位ゆれ (`10MB` / `10 MB` / `10メガバイト`)
- [ ] 引用するサービス名・API 名の表記ゆれ (`Cloudflare Pages` / `cloudflare pages`)
- [ ] サンプルコードと文章の食い違い (関数名・引数名・戻り値)
- [ ] バージョン番号や仕様の書き方が古くなっていないか (例: 廃止 API)
- [ ] リンク URL が `https://` で始まっているか / 相対リンクで内部記事を参照していないか
- [ ] `## 参照` のリンクテキストが「サイト名 + 種別」になっているか

## 公開準備チェックリスト (`/blog:publish`)

draft を外して公開可能にする最終整形。各項目で問題があればユーザーに確認してから進める。

1. `draft: true` をフロントマターから削除する (削除前にユーザーへ確認)
2. スクショ placeholder (`<!-- スクショ: ... -->` と `> 📸 ...`) の残存を検知。残っていれば「画像差し替え / 全削除 / そのまま (draft 復帰)」のいずれかをユーザーに選んでもらう
3. `## まとめ` と `## 参照` が末尾にあることを最終確認
4. `pnpm lint` を実行して prettier/eslint 違反が無いことを確認
5. 必要に応じて `pnpm check` を実行
6. `getAllPosts()` (`src/lib/utils/posts.ts`) が日付降順でソートする仕様なので、`date` を意図した日付に更新するかユーザーに確認

## やらないこと (アンチパターン)

- `## はじめに` セクションを作る
- `## 参照` を `## 参考リンク` `## References` `## 関連リンク` などにリネームする
- 公開記事に `draft: true` を残す
- 個人ブログや Qiita 記事を `## 参照` に入れる
- 装飾絵文字を 1 段落ごとに入れる
- 記事の途中で `---` を多用する
- フロントマターを 5 項目以上にする (`author` 等を追加しない、現状未使用なので)
- ファイル名に日付を含める (`2026-04-29-foo.md` のような形式は使わない)
